# study-quiz-manager 要件定義書

## 1. 概要

`study-quiz-manager` は、小学生・中学受験生向けの暗記学習を支援するローカル動作のWebアプリケーションである。塾や学校のテストなどで間違えた知識・用語・漢字表記を登録し、復習優先度に基づいて印刷用の小テストを作成する。小テスト実施後は、紙の答案を見ながら採点結果を入力し、その結果を次回以降の出題優先度や出題形式に反映する。

本アプリは、既存の `kanji_test_generator` をそのまま拡張するものではなく、国語・理科・社会の暗記系知識を扱う新規アプリとして設計する。PDF生成ではなく、ブラウザ印刷に適したHTML画面を出力する。

## 2. 基本方針

### 2.1 アプリの目的

このアプリの目的は、単に問題を作ることではなく、**間違えた知識を再出題し、定着まで管理すること**である。

主な目的は以下の通り。

1. 塾テストや家庭学習で間違えた知識項目を登録する。
2. しばらく復習していない項目や、ミスが多い項目を優先して小テストに出す。
3. 国語・理科・社会の暗記系知識を対象にする。
4. 漢字や用語の表記練習を兼ねるため、紙に印刷して手書きで解答する。
5. 採点結果を入力し、次回の出題優先度と出題形式に反映する。

### 2.2 対象科目

対象科目は以下とする。

- 国語
- 理科
- 社会

算数は当面対象外とする。算数の公式や単位換算などを将来扱う可能性はあるが、MVPでは含めない。

### 2.3 外部通信について

このアプリは、原則として外部通信を行わない。

- クラウド同期は行わない。
- ユーザー認証は行わない。
- メール通知やプッシュ通知は行わない。
- LLM API連携は行わない。
- 登録データはローカル環境内に保存する。

将来的にOCRやCodex等を使った登録補助を検討する場合でも、このアプリ本体は外部通信を前提にしない。外部ツールで抽出したデータを、CSVやJSONなどのローカルファイル経由で取り込む方式を優先する。

## 3. 用語定義

| 用語 | 意味 |
|---|---|
| 知識項目 | 復習対象となる用語・人物・漢字・語句・知識 |
| 小テスト | 登録済みの知識項目から作成する印刷用テスト |
| 問題用紙 | 子どもが解答するための印刷ページ |
| 解答用紙 | 保護者が採点するための正解表示ページ |
| 出題形式 | 一問一答、穴埋め、文中書き取りのいずれか |
| ミス種別 | 「字が違う」「違う言葉」の2種類 |
| 復習状態 | 最終出題日、次回出題日、正解数、ミス数などの状態 |

## 4. 主な利用フロー

### 4.1 知識項目の登録

塾のテストや家庭学習で間違えた問題を見て、保護者または学習者が知識項目を登録する。

登録時には、単なる「問題」ではなく、復習したい知識を中心に登録する。ただし、運用を重くしすぎないため、MVPでは「知識項目 + 代表問題」の形式で登録する。

登録例：

```text
科目：社会
カテゴリ：地理
単元：日本のすがた
タイトル：標準時子午線と明石市
問題文：日本の標準時子午線が通る市はどこですか。
正解：明石市
読み：あかしし
デフォルト出題形式：一問一答
```

### 4.2 小テスト作成

登録済みの知識項目から、復習優先度に基づいて小テストを作成する。

小テスト作成時には、以下を指定できる。

- 科目
- カテゴリ
- 問題数
- 出題対象
- 出題順

MVPでは、以下の操作を最小構成とする。

```text
科目を選ぶ
問題数を選ぶ
未復習・ミス優先で自動選出
印刷用ページを作る
```

### 4.3 印刷して手書き解答

小テストはPDFではなく、ブラウザ印刷用のHTMLページとして作成する。ChromeやEdgeの印刷機能を使い、A4用紙に印刷する。

紙に手書きで解答する理由は以下の通り。

- 国語や社会の漢字練習を兼ねるため。
- 表記ミスを確認するため。
- 実際のテストに近い形式にするため。
- 自動採点ではなく、親が答案を見て判断できるようにするため。

### 4.4 採点結果入力

小テスト実施後、紙の答案を見ながらアプリに採点結果を入力する。

採点は各問題につき、次の3つのボタンから選択する。

```text
○ 正解
△ 字が違う
× 違う言葉
```

内部的には、以下のように扱う。

| UI表示 | 内部結果 | ミス種別 |
|---|---|---|
| ○ 正解 | correct | なし |
| △ 字が違う | partial | wrong_character |
| × 違う言葉 | incorrect | wrong_term |

## 5. 出題形式

### 5.1 一問一答

知識そのものを直接問う形式。

例：

```text
794年に都を平安京に移した天皇は誰ですか。
答え：＿＿＿＿＿＿＿＿＿＿
```

主に、理科・社会の知識確認に使う。

### 5.2 穴埋め

文脈を提示し、空欄に答えを書かせる形式。読みは提示しない。

例：

```text
日本の標準時子午線は兵庫県（　　　）市を通ります。
答え：＿＿＿＿＿＿＿＿＿＿
```

主に、用語を文脈から思い出せるか確認するために使う。

### 5.3 文中書き取り

文脈の中に読みを提示し、正しい漢字・表記を書かせる形式。

例：

```text
日本の標準時子午線は兵庫県（あかし）市を通ります。
答え：＿＿＿＿＿＿＿＿＿＿
```

この形式は、国語の漢字問題にも、社会や理科の用語確認にも使う。

例：

```text
友人と意見を（こうかん）する。
答え：＿＿＿＿＿＿＿＿＿＿
```

```text
植物が光を受けてでんぷんを作るはたらきを（こうごうせい）といいます。
答え：＿＿＿＿＿＿＿＿＿＿
```

文中書き取りでは、読みだけを単独で提示するのではなく、必ず文脈を含む問題文として提示する。

## 6. ミス種別

ミス種別は多くしすぎず、以下の2種類に限定する。

### 6.1 違う言葉

正解とは別の用語・人物・語句を答えた状態。

例：

```text
正解：桓武天皇
解答：推古天皇
```

```text
正解：明石市
解答：神戸市
```

```text
正解：促成栽培
解答：抑制栽培
```

この場合、次回は一問一答または穴埋め形式で出題する。

### 6.2 字が違う

答えるべき対象はおおむね分かっているが、漢字・文字・表記が違う状態。

例：

```text
正解：桓武天皇
解答：桓文天皇
```

```text
正解：墾田永年私財法
解答：懇田永年私財法
```

```text
正解：促成栽培
解答：即成栽培
```

この場合、次回は文中書き取り形式で出題する。

### 6.3 迷った場合の判断基準

迷った場合は、「次に何を練習させたいか」で選ぶ。

| 次に練習させたいこと | 選ぶミス種別 |
|---|---|
| 正しい用語を思い出させたい | 違う言葉 |
| 正しい漢字・表記を書かせたい | 字が違う |

## 7. 復習優先度

### 7.1 基本方針

小テスト作成時には、以下の項目を優先する。

- 次回出題日を過ぎている項目
- しばらく復習していない項目
- ミスが正解数より多い項目
- 直近で間違えた項目
- 重要度が高い項目

### 7.2 次回出題日の更新ルール

採点結果に応じて、次回出題日を更新する。

| 採点結果 | 次回出題日 | 備考 |
|---|---|---|
| ○ 正解 | 7日後 | 連続正解数を増やす |
| △ 字が違う | 3日後 | 次回は文中書き取りを優先 |
| × 違う言葉 | 1日後 | 次回は一問一答または穴埋めを優先 |

連続正解が増えた場合は、出題間隔を延ばす。

例：

| 状態 | 次回出題間隔 |
|---|---|
| 正解1回 | 7日後 |
| 連続正解2回 | 14日後 |
| 連続正解3回 | 30日後 |
| さらに安定 | 定着扱い |

定着扱いの項目は通常の出題候補から外してよい。ただし、将来的には定着確認として低頻度で出題できるようにしてもよい。

### 7.3 優先度スコア

候補が多い場合は、以下のような優先度スコアで並べる。

```text
優先度 =
期限超過日数
+ max(0, ミス回数 - 正解回数) × 3
+ 直近ミスボーナス
+ 重要度
```

考え方：

- 復習予定日を過ぎたものほど優先する。
- ミス回数が正解回数を上回る項目ほど優先する。
- 直近で間違えた項目は優先する。
- 入試頻出など重要度が高いものは優先する。

「連続正解数」だけで判断すると、1回間違えた瞬間に優先度が不自然に跳ね上がる可能性があるため、累積の `ミス回数 - 正解回数` を使って苦手度を表す。

## 8. データモデル案

### 8.1 StudyItem

復習対象となる知識項目。

```ts
type Subject = 'japanese' | 'science' | 'social';

type QuestionType =
  | 'short_answer'
  | 'fill_blank'
  | 'contextual_writing';

type StudyItem = {
  id: string;
  subject: Subject;
  category: string;
  unit?: string;

  title: string;
  questionText: string;
  answer: string;
  reading?: string;
  note?: string;

  defaultQuestionType: QuestionType;
  importance: 1 | 2 | 3;

  status: 'active' | 'mastered' | 'paused';

  createdAt: string;
  updatedAt: string;
};
```

### 8.2 AnswerRecord

採点結果の履歴。

```ts
type Result = 'correct' | 'partial' | 'incorrect';

type MistakeType =
  | 'wrong_character'
  | 'wrong_term';

type AnswerRecord = {
  id: string;
  studyItemId: string;
  quizId: string;
  answeredAt: string;

  result: Result;
  mistakeType?: MistakeType;

  questionTypeUsed: QuestionType;
  memo?: string;
};
```

### 8.3 ReviewState

復習状態。

```ts
type ReviewState = {
  studyItemId: string;

  lastReviewedAt?: string;
  nextReviewDate?: string;

  correctCount: number;
  mistakeCount: number;
  consecutiveCorrectCount: number;

  lastResult?: Result;
  lastMistakeType?: MistakeType;
};
```

### 8.4 Quiz

小テスト単位の履歴。

```ts
type Quiz = {
  id: string;
  createdAt: string;
  title: string;

  subjectFilter?: Subject[];
  categoryFilter?: string[];
  itemIds: string[];

  printedAt?: string;
  scoredAt?: string;
};
```

## 9. 画面要件

### 9.1 知識項目一覧画面

登録済みの知識項目を一覧表示する。

必要な表示項目：

- 科目
- カテゴリ
- タイトル
- 正解
- 次回出題日
- 状態
- 最終結果

必要な操作：

- 新規登録
- 編集
- 削除
- 検索
- 科目・カテゴリで絞り込み

### 9.2 知識項目登録・編集画面

知識項目を登録・編集する。

入力項目：

- 科目
- カテゴリ
- 単元
- タイトル
- 問題文
- 正解
- 読み
- デフォルト出題形式
- 重要度
- メモ

文中書き取りの場合は、問題文プレビューを表示する。

### 9.3 小テスト作成画面

小テスト作成条件を指定する。

入力項目：

- 科目
- カテゴリ
- 問題数
- 出題対象
- 出題順

出力：

- 問題用紙プレビュー
- 解答用紙プレビュー
- 印刷ボタン

### 9.4 印刷用画面

ブラウザ印刷に適した画面を表示する。

要件：

- A4縦を前提にする。
- 問題番号を表示する。
- 手書き解答欄を十分に確保する。
- 問題用紙には正解を表示しない。
- 解答用紙には正解とメモを表示する。
- 印刷時には操作ボタンやナビゲーションを非表示にする。

### 9.5 採点入力画面

作成済み小テストに対して、採点結果を入力する。

各問題に以下のボタンを表示する。

```text
○ 正解
△ 字が違う
× 違う言葉
```

入力後、復習状態を更新する。

## 10. 技術方針

### 10.1 推奨技術スタック

`study-task-manager` の実績を参考に、以下を基本候補とする。

- React
- TypeScript
- Vite
- Tailwind CSS
- React Router
- Vitest
- Testing Library

### 10.2 データ保存

MVPではローカル保存を前提とする。

候補：

- localStorage
- IndexedDB

単純なMVPでは localStorage でもよいが、知識項目数が増える可能性を考えると IndexedDB も候補とする。

外部DBやクラウドDBは使用しない。

### 10.3 レイヤー構成

以下のようにレイヤーを分ける。

```text
src/
  app/        # 画面・ルーティング
  domain/     # ドメインモデル・復習ロジック・出題ロジック
  infra/      # localStorage / IndexedDB などの保存処理
  ui/         # 共通UI部品
  test/       # テスト共通設定
```

ドメインロジックをUIに直接書かない。特に以下は domain 層に置く。

- 優先度計算
- 次回出題日更新
- 出題形式決定
- 採点結果の反映
- 小テスト候補選出

## 11. MVP範囲

MVPでは以下を実装対象とする。

1. 知識項目の登録・編集・削除。
2. 国語・理科・社会の科目分類。
3. カテゴリ・単元による分類。
4. 一問一答・穴埋め・文中書き取りの3形式。
5. 未復習・ミス優先による小テスト作成。
6. ブラウザ印刷用の問題用紙・解答用紙。
7. 採点結果入力。
8. ○正解、△字が違う、×違う言葉の3分類。
9. 採点結果に応じた次回出題日更新。
10. ローカル保存。
11. バックアップ用のエクスポート・インポート。

## 12. MVPではやらないこと

以下はMVPでは実装しない。

- クラウド同期
- ユーザー認証
- マルチユーザー対応
- 統計ダッシュボード
- メール通知
- プッシュ通知
- LLM連携
- 自動採点
- PDF直接生成
- 複数語句の同時書き取り
- 算数対応
- 外部API通信

## 13. 今後の拡張案

### 13.1 カテゴリ・単元のカスタマイズ

ユーザーが任意のカテゴリや単元を追加・編集できるようにする。

例：

```text
社会 > 歴史 > 平安時代
社会 > 地理 > 日本の気候
理科 > 生物 > 植物
国語 > 語彙 > 慣用句
```

MVPでもカテゴリ・単元の入力欄は持たせるが、将来的には候補管理画面を作り、入力ミスや表記ゆれを防ぐ。

### 13.2 外部データの取り込み

将来的には、塾の試験プリントや復習メモから、誤答項目を効率よく登録できるようにしたい。

ただし、アプリ本体は外部通信を行わない方針とする。

想定する流れ：

1. Codex等の外部作業環境で、試験プリント画像やPDFをOCR処理する。
2. 誤答した問題・正解・読み・科目・カテゴリを抽出する。
3. 抽出結果をCSVまたはJSONとしてローカルファイルに保存する。
4. `study-quiz-manager` にそのファイルをインポートする。
5. ユーザーが内容を確認・修正してから登録する。

重要な方針：

- 自動登録ではなく、登録前に必ず確認画面を出す。
- OCR結果は誤認識があり得るため、正解・読み・問題文を編集できるようにする。
- インポート機能は、既存データの上書きではなく追加登録を基本とする。
- 重複候補がある場合は警告する。
- アプリ本体から外部OCRサービスやLLM APIを直接呼び出さない。

## 14. 非機能要件

### 14.1 ローカル完結

アプリはローカル環境で完結する。

- データはユーザー端末内に保存する。
- 外部サーバーへ送信しない。
- 認証を必要としない。
- 家庭内での単一ユーザー利用を前提とする。

### 14.2 印刷品質

ブラウザ印刷で使うため、印刷時の見やすさを重視する。

- A4縦向きで印刷しやすいこと。
- 手書き解答欄が十分にあること。
- 余白が適切であること。
- 画面用UIは印刷時に非表示になること。
- 解答用紙を別ページとして印刷できること。

### 14.3 保守性

CodexなどのLLMに実装を任せることを想定し、以下を守る。

- ドメインロジックとUIを分離する。
- 型定義を明確にする。
- 優先度計算や復習日更新にはユニットテストを書く。
- 保存処理は infra 層に閉じ込める。
- 画面コンポーネントに複雑な計算を書かない。

## 15. Codex向け実装メモ

最初に実装する順番は以下を推奨する。

1. プロジェクト初期化。
2. 型定義の作成。
3. localStorageまたはIndexedDB保存層の作成。
4. 復習状態更新ロジックの実装。
5. 優先度計算ロジックの実装。
6. 小テスト候補選出ロジックの実装。
7. 知識項目登録画面の実装。
8. 小テスト作成画面の実装。
9. 印刷用画面の実装。
10. 採点入力画面の実装。
11. エクスポート・インポートの実装。
12. テスト整備。

特にテストすべきロジック：

- 正解時に次回出題日が延びること。
- 字が違う時に次回出題形式が文中書き取りになること。
- 違う言葉の時に次回出題形式が一問一答または穴埋めになること。
- `max(0, ミス回数 - 正解回数)` により苦手度が計算されること。
- `nextReviewDate` が今日以前の項目が優先されること。
- `mastered` や `paused` の項目が通常出題から除外されること。

## 16. まとめ

`study-quiz-manager` は、国語・理科・社会の暗記系知識を登録し、紙に印刷して手書きで復習するためのローカルWebアプリである。

中心となる価値は以下である。

- 間違えた知識項目を登録する。
- 復習すべき時期の項目を自動で選ぶ。
- 漢字・表記ミスに応じて文中書き取り形式で再出題する。
- 別の言葉を答えた場合は一問一答や穴埋めで再確認する。
- 外部通信を行わず、家庭内・ローカル環境で完結する。

MVPでは、余計な統計やクラウド機能は入れず、登録・出題・印刷・採点・復習更新の基本サイクルを確実に動かすことを最優先とする。
